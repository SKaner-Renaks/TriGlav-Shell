# ТЕХНИЧЕСКОЕ ОПИСАНИЕ ПРОЕКТА: СЕРВИСНЫЙ МОДУЛЬ ОБНОВЛЕНИЯ  «TriGlav Shell»

## 1. Архитектура окружения и структура репозитория
* **Тип модуля:** Сервисный (`service`), папка имеет префикс `_` (например, `_updater`).
* **Структура данных в удаленном репозитории (GitHub):**
* **Ядро (Shell):** Тело основного скрипта и папка `_Data` с содержимым в корне репозитория.
* **Модули:** Находятся в папке `_module`. Каждый модуль — это отдельная подпапка.
* *Логика развертывания:* Сама папка `_module` в корне целевой системы создается (если её нет), но её общее содержимое целиком не скачивается — скачиваются только конкретные подпапки выбранных модулей.

## 2. Интерфейс пользователя (UI/UX) и жизненный цикл состояний

Для минимизации ошибок и конфликтов данных интерфейс работает по строгому триггерному принципу.

### Начальное состояние (при открытии модуля):

* Верхняя панель управления репозиториями активна.
* **Группа 1 (Установленные объекты)** — полностью пуста.
* **Группа 2 (Новые модули)** — полностью пуста.
* В обеих группах предусмотрено поле интерактивного текстового поиска (работает после заполнения таблиц).

### Панель управления репозиториями:

* Текстовое поле ввода URL (по умолчанию подставляется `https://github.com/SKaner-Renaks/TriGlav-Shell`).
* Выпадающий список альтернативных ссылок (из локального файла конфигурации в корне модуля обновления).
* Кнопка **«Get» / «Запрос»**.

### Активация интерфейса (Нажатие кнопки «Get»):

1. Показывается модальное окно с анимацией и текстовым статусом проверки сети и чтения манифестов.
2. Если репозиторий недоступен, выводится ошибка с предложением выбрать другую ссылку.
3. При успешном ответе происходит сканирование манифестов (в _data для shell и в подпапках папки `_module` для модулей) и **одновременное заполнение обеих таблиц**:

* **Таблица 1 (Установленные объекты):**
* *Сортировка:* 1. `shell` (ядро), 2. Сервисные модули `service`, 3. Обычные модули `usual`, 4. Игровые модули (`game`).
* *Колонки:* Чек-бокс, Type, Status (on/off), Name, Title, Current Ver, Repo Ver (заполняется из репозитория).


* **Таблица 2 (Новые модули):**
* *Колонки:* Чек-бокс, Type, Name, Title, Description, Repo Ver.

## 3. Логика выполнения действий и безопасность

* **Правило эксклюзивности:** За один раз пользователь может запустить **только одно** действие: либо *Обновление*, либо *Удаление*, либо *Установку*. Одновременное выполнение разных операций заблокировано.
* После завершения любого действия (или при необходимости сменить тип операции) пользователю необходимо **снова нажать кнопку «Get»**, чтобы модуль заново опросил систему, актуализировал статусы версий и перестроил таблицы.
* **Ограничения безопасности в Таблице 1:**
* `shell` удалить невозможно (чек-бокс удаления заблокирован).
* Сервисные модули (`_`) удаляются только после дополнительного подтверждения в чек-боксе (аналог логики `_module_manager`).
* При выборе `shell` или сервисных модулей для обновления выводится предупреждение: *«Внимание! Это действие приведет к полному перезапуску TriGlav Shell»*.

## 4. Алгоритмы процессов обновления и установки

При запуске «Обновления» или «Установки» модуль всегда **первым делом очищает подпапку `_updater/DownLoad/**` от старых файлов. Скачивание всегда происходит в эту папку, на экране отображается прогресс-бар с именем текущего объекта.

### Сценарий 1: Онлайн-обновление (Обычные и Игровые модули)

1. Объект скачивается в `DownLoad/`.
2. Shell отправляет команду на остановку веб-сервера конкретного модуля.
3. Папка модуля заменяется файлами из `DownLoad/`.
4. Shell снова запускает веб-сервер обновленного модуля. Процесс идет без остановки ядра.

### Сценарий 2: Офлайн-обновление (Shell и/или Сервисные модули)

*Решает проблему блокировки занятых файлов в Windows.*

1. Объекты скачиваются в `DownLoad/`. При обновлении `shell` проверяется чек-бокс пользователя *«Не перезаписывать конфигурационный файл»*. Если он нажат, файл конфигурации исключается из замены.
2. Модуль запускает автономный скрипт `runtime_updater.py` как **отсоединенный процесс (detached)**.
3. Основной Shell и все модули полностью выключаются, **освобождая единый порт**.
4. На фронтенде у пользователя запускается анимация ожидания на 10 секунд с последующим автообновлением страницы.
5. Скрипт `runtime_updater.py` перехватывает освободившийся порт и поднимает на нем временный Flask-интерфейс, транслируя ход замены файлов.
6. Поскольку процессы Shell мертвы, файлы не заблокированы ОС Windows. Скрипт копирует новые файлы ядра и сервисов на их места.
7. Скрипт запускает обновленный основной Shell, а сам завершает работу, отдавая ему порт. Через 10 секунд браузер пользователя перезагружает страницу и попадает в обновленную систему.

## 5. Разрешение зависимостей (Модули и Библиотеки Python)

#Lazurus **Архитектурное решение для управления зависимостями:**

> Чтобы не усложнять архитектуру апдейтера, логику проверки зависимостей нужно стандартизировать через манифесты модулей, а установку библиотек делегировать уже существующему у вас сервисному модулю.
> **Реализация:**
> 1. В манифест каждого модуля (`manifest.json`) добавляются два необязательных ключа:
> * `"dependencies_modules": []` — список имен сервисных или обычных модулей TriGlav, без которых данный модуль не запустится.
> * `"dependencies_pip": []` — список чистых Python-библиотек (как в `requirements.txt`).
> 
> 
> 2. При нажатии «Скачать и установить» во Второй таблице, модуль обновления проверяет ключ `"dependencies_modules"`. Если там указан модуль, которого нет в Системе, апдейтер автоматически добавляет его в очередь на скачивание и информирует пользователя: *«Для работы модуля X требуется модуль Y. Он будет установлен автоматически»*.
> 3. Что касается Python-библиотек: модуль обновления скачивает файлы. Затем, перед запуском веб-сервера нового модуля, апдейтер проверяет, установлен ли в системе ваш специализированный сервисный модуль для проверки библиотек.
> * *Если он установлен:* Апдейтер отправляет ему внутренний API-запрос на сканирование и установку pip-зависимостей для новой папки.
> * *Если его нет (или он не активирован):* Апдейтер выдает пользователю предупреждение в окне прогресса: *«Внимание! Для модуля требуются внешние библиотеки. Пожалуйста, убедитесь, что они установлены, или установите сервисный модуль менеджмента библиотек _deps_checker »*. В качестве альтернативы, апдейтер может сам выполнить скрытую системную команду `pip install -r` напрямую через `subprocess`, если в скачанной папке присутствует файл зависимостей. Это подстрахует систему в любом случае.
