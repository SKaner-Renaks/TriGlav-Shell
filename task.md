# ТЗ: Разработка веб-оболочки "ТриГлав" (TriGlav Shell)

## 1. Цель проекта
Создать управляющую веб-оболочку (Shell) на базе фреймворка **Flask** для Windows-окружения. Shell объединяет изолированные модули автоматизации (каждый из которых является отдельным Flask-приложением) в единую панель управления, полностью беря на себя авторизацию пользователей, динамическое управление процессами модулей и их конфигурациями.

## 2. Целевое окружение и Стек
- **Операционные системы:** Windows 10/11, Windows Server 2016+ (Linux НЕ поддерживается).
- **Стек разработки:** Python 3.10+, Flask, встроенный модуль `subprocess`, встроенный модуль `json`, `configparser`.
- **Интерфейс (UI):** Чистый HTML5/CSS3/JS, встроенный прямо в Python-скрипты через `render_template_string`. Цветовая палитра: Темная тема в стиле веб-консоли управления Proxmox.

---

## 3. Архитектура сети и Безопасность
Shell работает по принципу **Reverse Proxy доверенного периметра**.

1. **Единая авторизация:** У модулей полностью вырезается собственная логика авторизации (формы логина, сессии, LDAP-проверки). Авторизацию осуществляет исключительно Shell. Модули безоговорочно доверяют запросам, проксируемым через Shell.
2. **Режимы работы (изменяются в глобальном `config.cfg` Shell):**
   - **Боевой режим (Production):** Все модули принудительно запускаются на `127.0.0.1` (localhost). Внешний доступ к портам модулей закрыт. Доступ пользователя к модулю из внешней сети происходит только через Shell, который отображает модуль внутри `iframe`.
   - **Режим разработки (Development):** Модули могут запускаться и принимать соединения на любых IP-адресах (например, в серой сети `192.168.x.x`).

## 5. Требования к компонентам

### 5.1 Глобальная конфигурация Shell (`config.cfg`)

Формат: INI-файл с подробными комментариями и примерами для каждой строки:

* `port`: Порт веб-сервера Shell (по умолчанию `5000` или `8080`).
* `auth_mode`: Режим авторизации (`local`, `ad`, `both`, `none`).
* `environment`: Режим работы (`production`, `development`).
* `global_refresh_interval`: Частота обновления данных (в секундах), если не переопределена в модуле.
* `global_theme`: Тема оформления (`dark`, `light`).
* `allowed_ips`: Безопасность — список IP, подсетей или диапазонов, с которых разрешен доступ к Shell.

### 5.2 Манифест и Конфигурация модуля (`manifest.json`)

Каждый модуль хранит свои метаданные, текущие настройки и эталонные настройки в одном JSON-файле:

```json
{
  "name": "control",
  "title": "Панель управления",
  "version": "1.1",
  "description": "Управление файлами и PowerShell-консоль",
  "requires_admin": true,
  "current_port": 5002,
  "languages": ["ru", "en"],
  "current_settings": {
    "refresh_interval": 5,
    "log_level": "INFO"
  },
  "default_settings": {
    "refresh_interval": 5,
    "log_level": "INFO"
  }
}

```

### 5.3 Модуль шифрования (`encrypt.py`)

Содержит функции кодирования/декодирования данных.

* Основная задача: Чтение и запись зашифрованных данных в файл `care.env`.
* Реализовать поддержку динамического добавления/смены методов шифрования. Функции чтения/записи должны принимать параметр-идентификатор, указывающий, какую конкретно внутреннюю функцию кодирования использовать.
* При отсутствии `care.env` Shell создаёт его автоматически с дефолтными значениями `Admin / Admin`.

## 6. Алгоритмы и логика работы Shell

### 6.1 Логика Автообнаружения и Запуска процессов

1. При старте `main.py` (Shell) сканирует папку `_module/` на наличие подпапок.
2. В каждой подпапке ищется `manifest.json`. Если файл найден, модуль регистрируется.
3. **Выделение портов:** В боевом режиме Shell сканирует ОС на предмет свободных портов. Найденный свободный порт Shell принудительно записывает в поле `"current_port"` файла `manifest.json` данного модуля.
4. Shell запускает процесс каждого выбранного модуля через `subprocess.Popen`, передавая параметры IP (`127.0.0.1` или `0.0.0.0`) и выделенный порт в качестве аргументов командной строки или переменных окружения.
5. Если в манифесте `requires_admin = true` и пользователь дал явное согласие в UI, Shell инициирует запуск процесса с правами администратора Windows (используя логику на основе `ctypes.windll.shell32.ShellExecuteW` с дескриптором `runas`, аналогично примеру в `task_server.py`).

### 6.2 Авторизация (Логика из `task_server.py`)

* Перенести и адаптировать готовую логику авторизации по Active Directory (`ldap3`) и проверку вхождения пользователя в `AD_GROUP`.
* Защитить маршруты Shell декоратором `@login_required`. Полномочия (какому пользователю что отображать) распределять на основе групп AD или локальных ролей.

### 6.3 Локализация и Fallback-стратегия

* Настройки языка Shell хранятся в `_lang/`. Языки модуля — в его корне или его подпапке `_lang/`.
* **Алгоритм отката:** Если в глобальных настройках Shell выбран язык (например, `fr`), но в `manifest.json` модуля в массиве `"languages"` его нет, Shell применяет стратегию отката: пытается включить английский (`en`), если его нет — русский (`ru`), если языковой файл поврежден — отображает сырые ключи (raw-строки).

### 6.4 Управление настройками и Сброс (Reset to Default)

В меню Shell (кнопка «Шестерёнка») доступны функции:

* **Сброс настроек модуля:** Shell считывает блок `"default_settings"` из `manifest.json`, полностью перезаписывает им блок `"current_settings"`, после чего перезапускает процесс модуля через `subprocess`.
* **Сброс настроек консоли:** Восстановление дефолтного состояния `config.cfg`.

### 6.5 Веб-Интерфейс Shell (UI)

Интерфейс строится на базе расширенного примера вашего веб-шелл:

1. **Зона 1 (Верхняя панель):** Заголовок ("ТриГлав Shell"), версия, кнопка настроек ("Шестерёнка"), кнопка "Выход" (Logout), кнопка завершения работы сервера Shell (Shutdown) и текущее время.
2. **Рабочая зона:** Разделена на две вертикальные секции:
* **Зона 3 (Левая панель):** Динамический список подключенных модулей. На кнопке отображается `title` модуля, а под ним — краткое описание (`description`).
* **Зона 4 (Правая панель):** Окно `iframe`, в которое загружается `http://<ip>:<current_port>` выбранного модуля.


3. **Разделитель (Splitter):** Вертикальная граница между Зонами 3 и 4 интерактивно двигается мышью (JS Dragging). Конечная ширина левой панели в пикселях сохраняется на стороне клиента в `localStorage` браузера (чтобы не перезаписывать глобальный файл конфигурации при каждом движении мыши).

### 6.6 Логика Перезапуска веб-сервера Shell

* Если изменение настроек в UI требует перезапуска Shell, сервер выполняет программный рестарт.
* Фронтенд (JS) в этот момент перехватывает управление, блокирует экран красивой анимацией загрузки и начинает отправлять частые фоновые запросы (long-polling) на эндпоинт проверки здоровья. Как только Shell поднимается и отвечает кодом `200`, страница автоматически обновляется.
